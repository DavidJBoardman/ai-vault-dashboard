# 2D Geometry Analysis Report ‚Äî Step 8 (Analysis) / 2D Tab

**Status:** Design approved, awaiting implementation plan
**Date:** 2026-04-27
**Owner:** Yang Zhang

## Goal

Fill the currently empty `2D` tab on the **Analysis** page (`src/app/workflow/step-8-analysis/page.tsx`, line 129) with a self-contained report summarising the outputs of Step 4 (2D Geometry). The report is both a viewable summary on screen and an exportable artefact for archival/publication.

## Scope

- In: Step 4 outputs only (ROI/bay-proportion, cut-typology matching, reference points, bay-plan rendering).
- Out: Step 2 projection details, Step 3 segmentations, all 3D-side data (already covered by the existing 3D tab).
- No new backend endpoints; the report is a pure view over data already persisted in the Zustand store and project files.

## Deliverables

1. A `<Geometry2DReport>` component rendered inside `TabsContent value="2d"`.
2. Two export buttons in the report header:
   - **Download Bundle (.zip)** ‚Äî folder of `report.html` (self-contained, base64-inlined image and inline SVG), `bay-proportion.csv`, `cut-typology.csv`, `bay-plan.csv`, plus the projection PNG.
   - **Download PDF** ‚Äî vector PDF rendered via Electron `webContents.printToPDF` against the report DOM with a print stylesheet.
3. New optional `location?: string` field on the `Project` model + a small input on the project metadata UI so it can be populated before export.

## Report contents (4 sections + footer)

### 1. Project header
- Project title.
- Project location (new field ‚Äî see "Data model changes" below).
- Project ID (small/muted).
- Projection name.
- Generated-at timestamp.

### 2. Bay proportion
- "Best match" card: canonical label (e.g. *ad triangulum*), measured `vaultRatio`, error vs. canonical.
- Candidates table: rank ¬∑ label ¬∑ error ¬∑ delta-from-best. Sourced from `vaultRatioSuggestions`, sorted ascending by `err`.
- Empty state: "ROI bay-proportion analysis has not been run."

### 3. Cut typology summary
- One-line summary: *N bosses matched across M typology variants*.
- Match table: existing `templateMatchCsvRows` rendered with `templateMatchCsvColumns` as headers.
- **Pagination on screen**: ~10 rows per page, prev/next + page indicator. Striped, monospaced numerics.
- **In print/PDF**: print stylesheet expands the table to show all rows so the exported document isn't truncated.
- Empty state: "Template matching has not been run."

### 4. Bay plan preview (SVG)
- Inline `<svg viewBox=...>` sized to the saved ROI bounding box.
- Base layer: `<image>` of the projection clipped to the ROI rect.
- Overlay: one labelled circle per saved reference point. Letters **A, B, C, ‚Ä¶** in save order (matches Step 4 convention). White-stroked text for legibility on any background.
- Caption: "Bay plan preview ¬∑ {N} reference points".
- Empty state: "No reference points saved yet."

### Footer
Single muted line: *Generated by Vault Analyser v{version} ¬∑ {timestamp}*.

## Export bundle layout

```
{projectName}-2d-report-{YYYY-MM-DD}/
  report.html              (self-contained: inline CSS, base64 PNG, inline SVG, all tables)
  bay-proportion.csv
  cut-typology.csv
  bay-plan.csv             (reference points: letter, u, v)
  bay-plan.png             (rasterised bay-plan: projection clipped to ROI + labelled reference points)
```

`buildBundleZip` uses JSZip in the renderer. The HTML is rendered to a string with `react-dom/server` from the same component tree, so on-screen and exported HTML do not drift.

`bay-plan.png` is produced by serialising the in-DOM `<BayPlanSvg>` element to an SVG string, drawing it onto an offscreen `<canvas>` (with the projection image as the base layer + labelled reference-point overlays), and exporting `canvas.toBlob('image/png')`. This guarantees the standalone PNG matches the report's bay-plan figure exactly.

## PDF export

- Renderer triggers `window.electronAPI.report.printToPdf({ targetSelector: '#geometry2d-report' })`.
- Electron main creates a hidden `BrowserWindow`, loads the same React route in print mode (a query flag `?print=1` causes the report to render with all tables expanded and no chrome), waits for `did-finish-load`, calls `webContents.printToPDF({ printBackground: true, pageSize: 'A4' })`, returns the buffer.
- Renderer receives the buffer and triggers a save dialog via existing IPC.
- Vector output preserved (SVG bay plan stays vector; raster only on the projection image).

## Print stylesheet

Targets `@media print` and `body.print-mode` (the latter set when `?print=1`):
- Hides app chrome (sidebar, header, action buttons).
- Expands paginated cut-typology table to full rows.
- Forces page breaks before each major section heading.
- Removes hover states / background tints that print poorly.

## UX & visual style

The report should read like a printable academic figure, not an app dashboard. Concretely:

- **Typography** ‚Äî use the project's display font for section headings, sans-serif body, and tabular-figure numerics for all measurements. Generous line-height (~1.5).
- **Hierarchy** ‚Äî one large section heading per part (1‚Äì4), a short caption underneath each, then content. Section numbers prefix each heading so a reader can refer to "¬ß2 Bay proportion".
- **Whitespace** ‚Äî ample padding inside cards; clear separation between sections with a thin top border or extra vertical space rather than heavy dividers.
- **Tables** ‚Äî striped rows, right-aligned numerics with fixed decimal places, monospace for ratios/errors, units on column headers (not in every cell).
- **Colour** ‚Äî restrained: muted greys for body text, a single accent colour for the "best match" highlight and headings. No segmentation-style multi-colour palettes.
- **Best-match emphasis** ‚Äî the top bay-proportion card uses a subtle bordered/filled accent (consistent with the existing chord-method-result card on the 3D tab) so the key answer is immediately visible.
- **Bay-plan figure** ‚Äî labels (A, B, C‚Ä¶) rendered with a small white halo for legibility on any background; circle markers sized proportionally to the ROI.
- **Export buttons** ‚Äî placed top-right of the report, secondary-button style; not competing visually with the report content.
- **Consistency with existing UI** ‚Äî re-use shadcn `Card`, `Button`, `Table` primitives already used on the 3D tab so the two tabs feel like siblings.
- **Print parity** ‚Äî the print stylesheet keeps the same typography and spacing; only chrome and pagination controls are stripped.

## Empty / partial states

The report always renders all 4 section headings ‚Äî missing data shows a one-line muted message under the heading. This keeps the structure consistent across projects at different completion stages and makes "what's missing" visible at a glance.

## Out of scope (explicit non-goals)

- DOCX export (not requested; PDF + bundle covers archival).
- A separate "Reference points" table ‚Äî already covered by per-boss rows in cut-typology.
- ROI summary card ‚Äî measured ratio is already shown in the bay-proportion section.
- Methodology / parameters block ‚Äî can be added later if reproducibility becomes a requirement.
- User-editable interpretation notes ‚Äî out of scope.
- Backend report generation ‚Äî kept entirely in the renderer / Electron main.

## Risks & mitigations

- **Print/screen drift** ‚Äî single component tree rendered server-side for HTML export and triggered via `?print=1` for PDF; no separate templates.
- **Large bay-plan rasterisation bloating the bundle** ‚Äî the bay-plan PNG is rendered once at the ROI's natural pixel size; if it exceeds ~2 MB, downscale to 2048 px on the longer edge before embedding.
- **Pagination logic complexity** ‚Äî keep it dumb: client-side slice of the rows array, no virtualisation.
- **Project location field migration** ‚Äî making it optional avoids any migration needed for existing projects.

## Acceptance criteria

- The 2D tab on the Analysis page renders the four sections plus footer with real data for a project that has completed Step 4.

- "Download PDF" produces a multi-page A4 PDF with the bay-plan SVG rendered as vectors and the full cut-typology table not truncated.
- Empty states render without errors for projects with partial Step 4 data.
